# Данные: обучающие пары и eval-набор

Документ описывает, **как получить данные** для RuSemanticSearch: пары для дообучения bi-encoder и набор для честной оценки качества. Архитектура поиска — стандартная и несложная; узкое место проекта именно здесь.

Главный принцип:

> Обучающие данные могут быть большими и шумными (синтетика).
> Eval-данные должны быть маленькими, но проверенными руками.
> Эти два набора **никогда не пересекаются**.

---

## 1. Файлы данных и их схемы

```text
data/
  documents.jsonl     # корпус — то, по чему ищем
  train_pairs.jsonl   # обучающие пары (query -> positive [+ hard negatives])
  eval_queries.jsonl  # запросы для оценки
  qrels.jsonl         # разметка релевантности (что чему релевантно)
```

**documents.jsonl** — единственный источник правды о документах:

```json
{"id":"doc-1","title":"Semantic Search over Russian Texts","text":"Проект про семантический поиск русскоязычных документов.","source":"sample","tags":[],"url":""}
```

**train_pairs.jsonl** — храним **id**, а не текст (чтобы не дублировать корпус и не расходиться с ним):

```json
{"query":"поиск похожих документов на русском","positive_id":"doc-1","hard_negative_ids":["doc-7","doc-12"]}
```

**eval_queries.jsonl**:

```json
{"qid":"q-1","query":"хочу проект про поиск похожих документов на русском"}
```

**qrels.jsonl** — разметка в стиле TREC, с градацией (нужна для nDCG):

```json
{"qid":"q-1","doc_id":"doc-1","rel":2}
```

`rel`: `0` — нерелевантно, `1` — частично/смежно, `2` — точно релевантно.

## 2. Пайплайн A — обучающие пары

### Шаг 1. Генерация позитивов (синтетика)

Для каждого документа LLM генерирует N запросов, которыми пользователь мог бы его искать. В промпте важно требовать:

- короткие, естественные запросы на русском;
- разный стиль: и «ключевые слова», и полное предложение;
- перефразирование, синонимы; смесь ru/en, раз корпус смешанный;
- **не копировать слова из заголовка** дословно.

Результат: `(сгенерированный_запрос, исходный_документ)` = позитивная пара.

> ⚠️ Риск: синтетический запрос повторяет лексику документа → модель учится не смыслу, а совпадению слов. Лечится требованием перефразировать (выше) и hard-негативами (ниже).

### Шаг 2. Hard-негативы

Для каждой пары `(query, positive)` прогоняем BM25 (и/или базовую embedding-модель) и берём из топа документы, которые **не являются** позитивом. Они похожи по словам, но неправильные по смыслу — именно это учит модель отделять поверхностное сходство от смысла.

```text
query: поиск похожих проектов
positive: Hybrid semantic search over Russian text
hard negative: поиск похожих изображений   <- похоже по словам, не по смыслу
```

> ⚠️ Риск: ложный негатив — документ, который на самом деле релевантен (в корпусе бывают почти-дубли). Смягчение: пропускать топ-1–2 (вероятные настоящие позитивы), брать негативы с рангов ~5–50; при желании отфильтровать LLM-ой.

### Шаг 3. Сборка `train_pairs.jsonl`

`(query, positive_id, [hard_negative_ids])`.

Для `MultipleNegativesRankingLoss` достаточно даже `(query, positive)` — остальные примеры в батче автоматически работают как негативы (in-batch negatives). Явные hard-негативы добавляются дополнительной колонкой и усиливают обучение.

Объём: начни с нескольких сотен — пары тысяч пар. Больше ≠ лучше, если синтетика низкого качества.

## 3. Пайплайн B — eval-набор (самое важное)

Этот набор должен быть **надёжным**. Он меньше, но проверен руками, и по нему меряется всё.

### Шаг 1. Запросы (~50–100)

Лучший вариант — написать руками, добиваясь разнообразия: точное совпадение, перефраз, ru/en-смесь, «сложные» смысловые. Можно взять отложенную часть синтетики, но тогда перечитать и поправить каждый запрос. Запросы должны быть **не из обучающих**.

### Шаг 2. Пул кандидатов (pooling)

Размечать запрос против всего корпуса невозможно. Стандартный приём IR — *pooling*: для каждого запроса собираем объединение топ-k нескольких систем (BM25 + базовые эмбеддинги [+ позже дообученная]) → ~20–50 кандидатов на запрос. Размечаем только их.

> Почему так: полный корпус размечать нереально, а пул покрывает документы, которые вообще способна вытащить любая из систем. Если размечать только выдачу BM25 — оценка будет смещена в пользу лексического поиска.

### Шаг 3. Разметка (градуированная)

Каждой паре `(query, кандидат)` ставим `rel ∈ {0,1,2}` руками. 50 запросов × ~30 документов — посильно и куда убедительнее для портфолио, чем «на глаз». Можно сделать первый проход LLM-ой, но затем перепроверить вручную. Пишем в `qrels.jsonl`.

### Шаг 4. Заморозка

Eval-набор не подкручиваем под результаты (это утечка). Если нужно подбирать гиперпараметры — заведи **отдельный dev-набор**, а финальный test трогай один раз.

---

## 4. Как eval управляет сравнением

Считаем `Recall@K`, `MRR`, `nDCG@K`, `Precision@K` для всех систем по **одним и тем же** `qrels`:

```text
BM25 (с русской лемматизацией)
базовая embedding-модель (e5 и т.п., с префиксами query:/passage:)
дообученный bi-encoder
hybrid (RRF)
hybrid + cross-encoder reranker
```

Одинаковые запросы + одинаковая разметка + одинаковый K = честное сравнение «яблоки с яблоками».

---

## 5. Порядок сборки

```text
1. documents.jsonl        (корпус — от него зависит всё)
2. eval_queries + qrels   (чтобы мерить с первого дня)
3. BM25 baseline          -> замер
4. базовые эмбеддинги      -> замер
5. train_pairs + fine-tune -> замер
6. hybrid (RRF)           -> замер
7. reranker               -> замер
```

Eval строится **до** дообучения — тогда каждый шаг меряется против фиксированной цели, и видно, что реально помогло.

---

## 6. Чек-лист типичных ошибок

- Eval протёк в обучение (одни и те же запросы/пары) → завышенные метрики.
- Синтетические запросы копируют слова документа → модель учит лексику, а не смысл.
- Hard-негатив оказался релевантным (ложный негатив) → шум в обучении.
- Корпус слишком мал → метрики шумные, разница между методами незначима.
- e5 без префиксов `query:` / `passage:`.
- BM25 без русской лемматизации → нечестно слабый baseline.
- Размечен только пул BM25 → смещение оценки в пользу лексического поиска (нужен pooling из нескольких систем).
